# TWAG as a native app via Capacitor

## Context

The site works well as a web app, but a native shell adds three things the
browser can't:

1. **Lock-screen / banner push notifications** when an event the user RSVP'd
   for is about to start. The browser's Notifications API works only while
   the tab is open; native push works whether the app is foregrounded,
   backgrounded, or closed.
2. **Native geolocation prompt** (with persistent "Always" permission and
   battery-friendly accuracy modes), used to show the user's current location
   on the map. Web geolocation works but its permission UX inside a webview
   is fiddly and times out.
3. **App Store + Play Store distribution**, so attendees can install TWAG
   from the same place they install every other Tech Week app.

Capacitor is the right tool: it wraps the existing `docs/` HTML/CSS/JS
(unchanged) in a native iOS/Android shell and exposes the native APIs above
through a small set of plugins. **No rewrite.** Estimated effort: ~1 day to a
working TestFlight + Play Console internal build.

Branch: `natea/capacitor`, off `main`.

## Architecture: web shell pattern

```
┌─────────────────────────── native shell (iOS + Android) ──┐
│                                                            │
│   WKWebView / WebView                                      │
│   ┌────────────────────────────────────────────────────┐   │
│   │   docs/  (the existing site, unchanged)            │   │
│   │   - events_map_<city>.html                         │   │
│   │   - events_gallery_<city>.html                     │   │
│   │   - search.js / events_map.js / etc.               │   │
│   │   - <city>.geojson / <city>_gallery.json           │   │
│   │   - boston/thumbs/*.jpg / nyc/thumbs/*.jpg         │   │
│   │                                                    │   │
│   │   plus a tiny Capacitor adapter:                   │   │
│   │   - capacitor_bridge.js                            │   │
│   │     (wraps Push / Geolocation / Share plugins      │   │
│   │      behind feature-detected functions so the      │   │
│   │      web version still works unchanged)            │   │
│   └────────────────────────────────────────────────────┘   │
│                                                            │
│   Native plugins (loaded at runtime):                      │
│   - @capacitor/push-notifications  (local + remote)        │
│   - @capacitor/geolocation                                 │
│   - @capacitor/share                                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

Same `docs/` folder serves both **GitHub Pages** and the bundled-in **native
app**. Feature detection (`if (window.Capacitor)`) decides whether to call
plugin APIs or fall back to web APIs.

## Decisions to confirm before implementing

1. **Push: local-only or also remote?**
   - **Local** (recommended for v1): the app schedules reminders on the device
     via `LocalNotifications.schedule()` for events the user has chosen. No
     backend, no APNs/FCM setup, no privacy story.
   - **Remote**: needs APNs key + FCM service account + a server to push
     from. Lets us notify everyone about late-breaking changes, but requires
     a backend that doesn't exist yet.
2. **Which events to remind about?**
   - Once the "My Schedule" feature ships (already planned in
     `my-schedule.md`), schedule a local notification 15 min before every
     event the user added.
   - Alternative: a "🔔 Remind me" button on every event popup/tile that
     schedules a one-off notification.
   - Or both.
3. **App Store + Play Store readiness now, or build only?**
   - "Build only" = TestFlight + Play Console Internal Testing track (no
     review needed). Useful for our own phones.
   - "Public release" = submit to App Store + Play Store review (~3 days
     Apple, ~1 day Google). Needs screenshots, privacy nutrition labels,
     content rating, etc.
4. **App icon + splash screen**: regenerate from the existing favicon SVG
   via `@capacitor/assets` (recommended), or hand-design at 1024×1024?

- `ios/` and `android/` — generated by `npx cap add ios/android`, mostly
  committed but with the heavy build artifacts gitignored.

### New (under `docs/`)

- **`docs/capacitor_bridge.js`** — tiny adapter loaded by all 5 HTML pages
  *before* `events_map.js` / `events_gallery.js`. Exports a `twagNative`
  object with these methods, all feature-detected so they no-op on the web
  build:
  - `requestLocation()` → returns `[lon, lat]` or null. Wraps
    `@capacitor/geolocation` natively; falls back to
    `navigator.geolocation` on the web.
  - `scheduleEventReminder(event, minutesBefore = 15)` → schedules a local
    notification. Stores the scheduled-notification ID in `localStorage` so
    it can be cancelled if the user un-RSVPs.
  - `cancelEventReminder(eventId)` → cancels by stored ID.
  - `shareEvent(event)` → opens the native share sheet via
    `@capacitor/share` on native; copies a URL to clipboard on the web.
  - `isNative()` → true when running inside the Capacitor shell.

### Modified

- **`docs/events_map.js`** — when `twagNative.isNative()`, add a "Locate me"
  button (or use Mapbox's `GeolocateControl` with the Capacitor geolocation
  provider). Show the user's blue dot on the map.
- **`docs/events_map_sidebar.js`** + **`docs/events_gallery.js`** — add a
  `🔔 Remind me` button next to the RSVP button. Click → calls
  `twagNative.scheduleEventReminder(event)`. Visual toggle when scheduled.
- **`docs/events_map.js`** popup HTML — small `📤` icon next to RSVP that
  calls `twagNative.shareEvent(event)` on native, falls back to Web Share
  API on the web (modern browsers support it).
- **All 5 HTML pages** — `<script src="./capacitor_bridge.js"></script>`
  before the page-specific JS.

### Reused

- The existing geojson / gallery JSON / thumbnails — Capacitor copies all of
  `docs/` into the native bundle at build time.
- Mapbox token, OpenCage data, PostHog/Datadog wiring — works the same in
  the webview as in the browser.
- The existing search, scope toggle, date picker, bottom bar — no changes.

## Push notifications: design

### v1 plan (local only)

User flow:

1. User taps `🔔 Remind me` on an event card → bridge calls
   `LocalNotifications.requestPermissions()` (iOS shows the system prompt
   on first call).
2. On grant, bridge calls `LocalNotifications.schedule({...})` with:
   - `at`: event `start_iso` minus 15 minutes
   - `title`: event title
   - `body`: "Starts in 15 min at {venue}. Tap to view."
   - `extra`: `{ event_id, city, view: 'map' }`
   - `actionTypeId`: "OPEN_EVENT" (so tapping opens the right deep link)
3. Capacitor stores the scheduled `id` in `localStorage` keyed by `event_id`
   so the user can cancel later (un-RSVP).
4. When notification fires, OS shows banner/sound/badge. Tap → app opens
   to `events_map_<city>.html#date=YYYY-MM-DD&event=<event_id>` and the
   sidebar auto-selects the event.

Edge cases handled:
- Permission denied → fall back to an in-app "remind me" log; show a polite
  one-line nudge to enable notifications in Settings.
- Notification scheduled for an event that's already started → don't
  schedule; show a "starts in N min" inline instead.
- Time zones: `start_iso` is UTC; `LocalNotifications` accepts a `Date`;
  device's local clock handles the conversion.

### Future: remote push (out of scope for v1)

If we ever want "the venue changed" alerts pushed to everyone who RSVP'd,
we'd add:
- APNs auth key + bundle ID at Apple
- Firebase project + FCM credentials
- A tiny backend (Cloud Run / Vercel function) that maintains a registry
  of device tokens per event and pushes via APNs/FCM
- `@capacitor/push-notifications` registration on app launch

Skip for v1. Local notifications cover the "next event starting" use case
end-to-end with no backend.

## Geolocation: design

Three places to use it:

1. **Map "Locate me" button** — Mapbox's `GeolocateControl` already does
   this in the browser; on native we replace the underlying provider with
   the Capacitor plugin so we get the better native permission UX and
   "Always" / "While Using" choices.
2. **Sidebar sort: "Closest to me"** — once we have a location, an optional
   sort mode ranks visible-viewport events by distance from the user.
   Power users on the ground will love this. Not v1 critical.
3. **Walking-time estimates** — already in the My Schedule plan; could
   integrate user's current location as the starting point for the next
   event's walking ETA.

For v1, ship #1 only.

## Share sheet integration

One-line addition to the map popup and sidebar detail card:

```js
button.addEventListener('click', () => {
  twagNative.shareEvent({
    title: event.title,
    text: `${event.title} — ${event.start_time} at ${event.venue_name}`,
    url: event.rsvp_url,
  });
});
```

On native, opens the OS share sheet (AirDrop, Messages, Mail, Slack, Notes,
"Save to Files," etc.). On web, falls back to `navigator.share()` (modern
browsers) → clipboard copy as final fallback.

## Out of scope (deliberate)

- Remote push notifications (no backend, deferred).
- Offline-first behavior. We could cache geojson + thumbs for offline use
  via a service worker (already PWA-friendly) but the live Mapbox tiles
  still need network.
- Apple Wallet / Google Wallet event passes. Cool idea, separate scope.
- iPad/Tablet layouts. The current responsive design works in landscape
  but isn't optimized.
- Telegram bot integration. The bot stays as-is; the native app is
  independent.
